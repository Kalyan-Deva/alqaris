DEVICE CONTEXT
Why It Matters

The BD Alaris 8015 is deployed in hospital ICUs and emergency wards worldwide to automatically deliver IV medications at precise rates. It connects to hospital Wi-Fi to receive drug library updates, sync with EHR systems, and transmit patient infusion data to central monitoring.

This network connection is what makes it dangerous — credentials stored on the device become a key to the entire hospital clinical network (MedNet VLAN).

⚠ BD confirmed they will not release a software patch for either vulnerability. All mitigations are operational — not technical.

CVE-2016-8375

CompactFlash
Card Swap Attack

AFFECTS: BD ALARIS 8015 v9.7 + 8000 PC UNIT
01
Gain Physical Access
Attacker approaches pump at bedside, in a hallway, storage closet, or during device maintenance. Average hospital has 200+ infusion pumps, many unattended. No authentication or badge required — just presence.
02
Swap the CompactFlash Card
Eject the external Wi-Fi CF card from its slot. Insert a pre-prepared malicious logging card. Externally identical. Process: 8–12 seconds. No tools, no damage, no visible sign of tampering.
03
Device Writes Credentials to Card
On reconnect or power-cycle, the Alaris 8015 writes hospital Wi-Fi SSID and PSK to the CF card in complete plaintext — no encryption, no key derivation, no obfuscation of any kind. The malicious card logs everything.
04
Retrieve Card → Network Access
Return on a second pass, retrieve the card, and mount it on a laptop. Run cat cf_dump.log | grep -E 'ssid|psk'. Hospital Wi-Fi password appears in plaintext. Join MedNet from any Wi-Fi coverage location — even the car park.

CVE-2016-9355

Internal Flash Memory
Extraction Attack

AFFECTS: BD ALARIS 8015 FIRMWARE ≤ v9.5
01
Disassemble the Device
Remove 4× Phillips PH1 screws from back panel. Open casing — no tamper-evident hardware in baseline configuration. BD's only mitigation is applying tamper stickers optionally, after deployment. Photograph PCB for documentation.
02
Remove Flash Chip from ZIF Socket
The internal flash chip sits in a zero insertion force (ZIF) socket — not soldered to the PCB. Lift the retention lever and slide the chip out. Takes under 5 seconds. No heat, no desoldering, no PCB damage whatsoever.
03
Read Chip Offsite — Zero Detection Risk
Unlike CVE-8375, the entire extraction happens away from the hospital. Insert chip into CH341A programmer ZIF socket at attacker's lab. Run: sudo flashrom -p ch341a_spi -r firmware.bin. Full dump in 3–8 minutes. Zero detection risk.
04
Extract Credentials with binwalk + HxD
Run binwalk -e firmware.bin to extract filesystem. Load in HxD and search for ASCII string ssid. Credentials found at offset 0x4A800 in plaintext. Full hospital Wi-Fi access obtained.

Internal Flash Chip

Removable module in ZIF socket on main PCB. Contains full device configuration including Wi-Fi credentials. No soldering required for removal.

CH341A Flash Programmer (~$10)

USB-connected flash reader. Supports SPI/I2C chips. Reads full firmware image to binary file. Works on any laptop with Linux/Windows.

$ binwalk --extract firmware.bin

// PLAINTEXT at offset 0x4A800 — no encryption

08 // DEFENSE ANALYSIS

DEFENSE GAP ANALYSIS

BD's four published mitigations (NHS Advisory CC-3795) — rated for effectiveness
Dedicated Medical Device VLANs
PARTIAL
BD says: "Isolate Alaris devices on a dedicated VLAN, separate from general hospital IT network."
Reduces post-exploitation blast radius but does NOT prevent credential extraction. The PSK is still stored in plaintext on the device. Once stolen, the attacker joins the VLAN regardless of segmentation. Segmentation ≠ authentication security. A determined attacker bypasses this completely.
Tamper-Proof Seals on Devices
WEAK
BD says: "Apply tamper-evident stickers to CF slot and chassis screws to detect physical tampering."
Stickers are cosmetic security. A sophisticated attacker can remove, replace, or bypass tamper seals with a heat gun and replacement stickers. Seals do not prevent extraction — they only flag that it happened, after the fact. CVE-8375 can be executed in 10 seconds with no seal damage. This is not a technical control.
Update to Non-EOL Firmware (≥ v9.19)
INCOMPLETE
BD says: "Ensure all devices run firmware ≥ v9.19. Versions below this are end-of-life and unsupported."
BD never confirmed that v9.19 encrypts credentials. They only said older versions are EOL. Neither CVE has a formally patched firmware version. Updating to v9.19 removes the device from the supported scope, but BD has not confirmed it eliminates the root vulnerability — unencrypted credential storage.
Physical Access Controls
NECESSARY
BD says: (Implied) Control physical access through standard hospital security measures.
This is the only genuinely meaningful mitigation. Both CVEs require hands-on device access — preventing physical contact prevents both exploits entirely. However, this is operationally very difficult in busy ICU environments where pumps are frequently moved, transported, and left unattended at bedsides. Not a scalable solution.

09 // PROPOSED IMPROVEMENTS

WHAT BD SHOULD HAVE DONE

Six concrete technical improvements — with feasibility and cost estimates
01
HIGH FEASIBILITY
AES-256 Credential Encryption
Encrypt Wi-Fi PSK using AES-256 with a key derived from device serial number + hardware UID at manufacturing. Even if flash is extracted, the PSK is unreadable without the device-specific key. Cost: minimal firmware change. No hardware modification needed.
02
MED FEASIBILITY
Hardware Security Element
Add an ATECC608 secure MCU (~$0.50/unit) to enforce hardware-level flash encryption. All sensitive data encrypted at hardware level — CH341A extraction yields only ciphertext. Industry standard in modern IoT devices. Requires PCB revision.
03
HIGH FEASIBILITY
802.1X EAP — No More PSK
Replace pre-shared key Wi-Fi with WPA3-Enterprise + 802.1X certificate-based authentication. Each device gets a unique certificate from a hospital CA. Theft of one certificate doesn't compromise the entire network. Standard practice in enterprise environments.
04
MED FEASIBILITY
Hardware Tamper Detection Switch
Install a microswitch behind the chassis screws. On tamper detection: log event to central audit trail, disable Wi-Fi connectivity, alert central monitoring station. Requires hardware revision but goes far beyond cosmetic sticker mitigation. Detects both CVE attack vectors.
05
HIGH FEASIBILITY
Secure Boot + Firmware Signing
Implement cryptographic firmware signature verification at boot. Only BD-signed firmware can execute. Prevents re-flashing flash chip with malicious firmware. Also prevents credential injection attacks. Already standard practice in FDA-cleared medical devices post-2021 guidance.
06
MED FEASIBILITY
Credential Rotation + Expiry
Rotate Wi-Fi credentials every 24–72 hours via an encrypted server push. Stolen PSK expires before the attacker can operationalize it. Requires central credential management infrastructure in the hospital, but eliminates the static PSK problem entirely across all devices.
